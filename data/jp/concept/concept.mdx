# Spear コンセプト

世の中には、[Next.js](https://nextjs.org/)や[Astro](https://astro.build/)・[Gatsby](https://www.gatsbyjs.com/).のような SSG ツールがたくさん存在します。

Spear もまた以下の特徴を持つ SSG ツールです :

- ヘッドレス CMS と接続が可能
- 軽量フレームワーク
- HTML/CSS だけ知っていれば他の知識は不要
- SEO を最初からサポート

## ウェブサイトの歴史を振り返る

Web 技術の歴史はブラウザ、JavaScript 仕様、HTML/CSS 仕様との歴史です。  
また、開発者がウェブの互換性と戦った歴史でもあります。

![Web Compatibility](/images/concept/compatibility-issue.jpeg)

### 互換性のジレンマ

ウェブの標準化以前、ウェブの構築はとても原始的でした。テキストエディタで HTML を書いて、簡単な CSS を使ってスタイルを当てる。
もし、ブラウザの互換性の問題が存在しなければ、標準仕様は生まれなかったでしょう。しかし現実はそうではなかったです。ブラウザの中には JavaScript をサポートしなかったり、特定のスタイル・HTML 要素などをサポートしないものもありました。そこで W3C が設立され、ウェブの標準仕様を発行してきました。

W3C はウェブ技術を進化させ、多くの仕様書を発行しました。しかし、互換性の問題は存在し続けました。特に InternetExplorerはひどかった。
そこで互換性問題を解決する手段として、ライブラリを利用するようになりました。よく知られているのはブートストラップです。

ウェブ上でのブラウザ互換性問題は大きな問題となりました。ウェブ仕様は早く書かれますが、いくつかのブラウザベンダは会社の戦略上ブラウザプロジェクトに人的リソースを追加できないところもありました。
この問題は現在も存在し続けます。しかし、ブラウザベンダは「ウェブブラウザはプラットフォームである」ということを学びました。彼らは互換性問題にとても敏感になりました。例えば、仕様提案書を書く際に彼らはすべてのブラウザベンダの反応を書いて、それをリリースする指標としています。

JavaScript の視点から言うと、JavaScript も同じ問題を抱えています。
しかし、JavaScript はユーザーエージェントによる動的な実装の変更や、ポリフィルと言った技術が使えました。

### 現代のウェブ互換問題を考えてみる

上記で書いたとおり、ブラウザベンダはウェブ互換性を考慮しています。そのため、以前に比べて互換性を注意深く気をつける必要がなくなりました。しかし、まだ少し気にする必要があります。

私達が考えていることは互換性ライブラリ・ポリフィル・ロールアップライブラリなど使うことなく HTML と CSS の標準仕様でウェブサイトを作れるのではないかということです。
ブラウザはプラットフォームです。歴史的にプラットフォームは必要な機能を提供し、便利なツールはサードベンダーが目まぐるしくリリースしています。  
最も重要なことはウェブサイトを構築するための機能は十分にリリースされているということです。別な言い方だと、HTML/CSS/JavaScript だけで開発者はウェブサイトを構築できるということです。

## クリエイターに寄り添う

今世の中の半分近くのウェブサイトは WordPress で作られています[1]。しかしながら、堅実なクリエイターは WordPress に問題があることをわかっています。

- 更新(脆弱性)
WordPress は多くのプラグインがあり、このプラグインはセキュリティリスクがある場合があります。運営者は毎日プラグイン更新をチェックする必要があります。でなければ WordPress 向けのサードパーティにお金を払うことになります。
- スケーラビリティ
WordPress サイトをスケールさせるサービスは多いです。これは WordPress のスケールアップが簡単ではないことを示しています。 WrodPress は PHP とデータベースの動くサーバーが必要で、サーバーマシンを増やすことはたやすくありません(我々は LAMPP 時代を思い出す)。

### なんでクリエイターは SSG を使わないのか？

我々はなんでクリエイターが SSG を使わないか(日本国内で)アンケートを取りました。
多くのクリエイターはクライアント企業と契約していて、ウェブサイトを維持管理する予算が豊富にありません。(つまり、クライアント企業はメンテナンス費用をあまり意識していない)  

この結果、クリエイターは依存パッケージの更新やライブラリの破壊的変更に伴う修正が必要になるような、メンテナンスが必要なツールを使いたがらないようです。

## Spear の柱

1. 🧙HTML と CSS だけでウェブサイトを構築する
2. 🚀軽量フレームワーク
3. 🧐下位互換性

### HTML と CSS だけでウェブサイトを構築する

近年、フロント技術はとても複雑になってきました。  
JavaScript の基礎技術に加え、フレームワークのルールなどを学ぶ必要があります。

例えば、Vue / React のようなリアクティブなフレームワークを使う場合、リアクティブやイベントの飛ばし方などを知る必要があります。

[ウェブサイトを振り返る]章でも記載したとおり、我々は HTML / CSS / (時々 JavaScript) だけでウェブサイトを構築できると信じています。

我々の Spear は HTML と CSS の知識だけで構築可能です。🧙

このプロジェクトドキュメントは、JavaScript コードは 8% で、その多くが設定ファイルです。その他は highlight.js のコードです！

### 軽量フレームワーク

モダンなフレームワークは構築が複雑で、多くのスニペットが含まれます。
ビルド後のオブジェクトはとても巨大です。

Spear のゴールの1つが軽量で、オブジェクトビルドを減らすことです。

### 下位互換性

もちろん、下位互換性を常に意識して破壊的な変更はリリースしません。

下位互換性は、廃止プロセスがとても重要です。
開発者はライブラリをアップデートした後「has no function」エラーに出くわすことがあります。ライブラリは時々彼らが不要と思った機能を削除します。

Spear はそのような破壊的な変更をリリースすることがありません！
